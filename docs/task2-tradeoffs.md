# 任務 2：避免重複運算 — 缺點與權衡

本設計將**情感標籤**與**詞元（post_tokens）**在寫入或 backfill 時計算並持久化，儀表板請求改以 SQL 聚合為主。以下為主要代價與風險。

## 儲存成本

- `post_tokens` 會顯著放大資料庫體積：每篇貼文可能對應多筆 `(post_id, token, cnt)`，索引（例如 `token`、`post_id`）亦佔空間。
- `posts` 新增 `sentiment_*`、`nlp_version` 欄位，對超大表亦有邊際成本。

## 資料一致性

- 若未來提供「只改 `content`」的更新 API 而未同步重算 tokens／sentiment，儀表板聚合會與實際內容不一致。
- 建議約定：**內容變更即標記需重分析**，或刪除該篇的 `post_tokens` 並將 `nlp_version` 設為低於 `CurrentNLPVersion` 以觸發 backfill。

## 模型／規則更新

- `nlp/app.py` 的詞典或規則變更時，舊資料的情感標籤可能「語意過期」。程式以 **`nlp_version`（對應 `CurrentNLPVersion`）** 區分；版本升級後應執行 **`POST /admin/backfill`**（或等效批次作業）重算。
- 與一般快取相同：需營運上記得「版本號 + 重算」，否則優化會變成技術債。

## 寫入延遲與吞吐

- `POST /posts` 路徑會同步呼叫 NLP 服務並寫入 tokens，單筆延遲高於「只寫 DB」；高併發寫入時 NLP 可能成瓶頸。
- 緩解方向（本專題未強制實作）：非同步佇列、批量寫入後離線分析、或僅在儀表板首次查詢時 lazy 計算（取捨不同）。

## 語意與維護

- 關鍵字篩選仍以 **`content` 子字串（LIKE）** 與日期範圍為主；在極大資料量下，若缺少全文檢索（如 FTS5），篩選成本仍可能偏高。
- Go 與 Python 兩端若將來分岔情感邏輯，需以版本號與測試鎖定一致性。

## 小結

此優化以**空間換時間**、以**離線／寫入時計算**換**讀取路徑穩定延遲**；代價是儲存、版本治理與更新流程必須一併納入設計。
